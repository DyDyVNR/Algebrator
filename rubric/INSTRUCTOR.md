# CIT 128 Project: Student Directed Project Instructor Rubric

**Version:** 4/30/2024

## Rubric / Grading Scale

The instructor provided rubric is worth 150 points **may not be modified** by the student. It consists of requirements that all students must achieve regardless of individual project description or goals. This page contains the instructor rubric information. This is how the instructor will break down the point totals for grading.

Items listed on this page may not be used in the student rubric.

## Instructor Grading Rubric (150pt)

* **Update the main README.md with the following (3 points total)**
  * Update the Student Name, CRN, & Semester / Year. (1pt)
  * Update the Heading 1 title of the program (1pt)
  * Markdown file is properly formatted. (1pt)
* **Update assignment/1_PLANNING.md (8 points total)**
  * Update the Heading 2 title of the program (1pt)
  * Update the *Project Description* with what the project is about (2pt)
  * Update the *Project Selection* with the reason why you selected the project (2pt)
  * Update the *Project Timeline* with the proposed timeline using bullet format for when the student expects major milestone goals will be achieved each week. (2pt)
  * Markdown file is properly formatted. (1pt)
* **Update assignment/2_ANALYSIS.md (11 points total)**
  * Update the list of the functional and non-functional requirements in the space provided. Must have a minimum of 3 or more for each requirement. (6pt)
  * Select and update the *Student Selected SDLC Analysis Concept 1* (2pt)
  * Select and update the *Student Selected SDLC Analysis Concept 2* (2pt)
  * Markdown file is properly formatted. (1pt)
* **Update assignment/3_TESTS.md (17 points total)**
  * Write a test plan for how to fully test the program including a numbered step by step walk through for manual testing. If unit tests are provided, make sure to include that process in the test script with how to trigger the unit test. (6pt)
  * Expected input methods or input values to be used as part of the testing and what values will trigger errors are provided. (4pt)
  * Expected output as a result of interaction or processing given as part of the testing. (6pt)
  * Markdown on TESTS.md is properly formatted. (1pt)
* **Demonstration Video on assignment/3_TESTS.md (14 points total)**
  * Provide a formatted markdown link to a video recording demonstrating the working application. (2pt)
  * Video recording is a minimum of 3-5 minutes in length, but may be longer. (4pt)
  * Student verbally describes how to start and use the application. (4pt)
  * Student discusses the testing process as part of the project and the verbal description matches the written description. (4pt)
* **Update assignment/4_IMPLEMENT.md (3 points total)**
  * Provide instructions on how to execute the software. (1pt)
  * Provide instructions on how to use the software. (1pt)
  * Markdown file is properly formatted. (1pt)
* **Update assignment/5_RETROSPECTIVE.md (3 points total)**
  * Discuss the success and improvements of the technical aspect of the project. (1pt)
  * Discuss how to improve yourself in the future. (1pt)
  * Markdown file is properly formatted. (1pt)
* **Update rubric/STUDENT.md (15 points total)**
  * A draft version of the Student Grading Rubric was updated by the Draft 1. (5pt)
  * Suggestions or final rubric approval from the instructor are merged via a GitHub Pull Request. (5pt)
  * Final student grading rubric grading descriptions are fully explained with points assigned to each point two weeks prior to the project due date. (4pt)
  * Markdown file is properly formatted. (1pt)
* **Programming of the Application (30 points total)**
  * GitHub Commits: Weekly programming commits showing progress. (6pt)
  * Write a minimum of 300 - 600 source lines of code (sloc) as shown in the GitHub repo. Provided markdown files, third-party modules, or externally generated files do not count towards the line count but may be used. (12pt)
  * Application is [documented using Docstrings](https://realpython.com/documenting-python-code/) appropriately. (3pt)
  * Comments are used to flag specific blocks of code. Comments used as Docstrings will receive no points for either section. (3pt)
  * Application code is well organized and has appropriate whitespace for readability. (6pt)
* **Structure of the Application Code (46 points total)**
  * The majority of the core application code written by the student uses Object-Oriented Programming classes and techniques. (17pt)
  * More than 1/2 of the minimum source code is an object-oriented class written by the student. (15pt)
  * The main program file uses functional or procedural programming techniques to start the application and instantiate objects from classes. (14pt)

## Additional Grading Notes

* Customers will not pay for programs that do not work, programs that do not run due to syntax errors will receive **0 Points**.
* All programs must have your Github Classroom URL submitted in Canvas via the assignment page. Unsubmitted Github repos or repos that do not use GitHub Classroom will receive **0 Points**.
* Programs that have been submitted and received a grade will not be regraded, unless the instructor makes a request for changes. If a program is eligible for regrading, it is the responsibility of the student to inform the instructor when the program is ready for regrading.
* Professional programmers are required to hit tough deadlines, late projects will loose **20 Points**.
* While automated unit tests may be used, manual testing of your code is also required. Hint, if you write a unit test, that counts towards the line count.
* Submitted code may be subject to inspection by the [Measure of Software Similarity (MOSS)](https://theory.stanford.edu/~aiken/moss/). to check for similarity in programs that may be related to plagiarism. It is up to the instructor to determine if plagiarism is occurring.
* Use of AI to assist your programming is prohibited. The goal is for you to learn how to program before learning AI-assisted programming. Projects that utilize AI to write code will receive **0 Points**.
* Programmers need to keep any confidential information secure and ensure that it is never published to GitHub. Confidential information published to GitHub will loose **5 Points**.
